# TDD implementation packet: selection vs attention colors

## 1. Goal

Make Web Cockpit color meaning intuitive: **teal = selection / primary action**, **mustard = attention / needs you**. Active project pills and the current bottom-nav page must use teal, not mustard. Attention badges and the "Needs you" section stay mustard.

## 2. Allowed files

**Tests**
- `crates/ajax-web/web/src/components/TaskList.test.ts`
- `crates/ajax-web/web/src/components/App.test.ts`

**Production**
- `crates/ajax-web/web/src/components/TaskList.svelte` (scoped styles only for pills/rows)
- `crates/ajax-web/web/src/styles.css` (bottom-nav active / aria-current)
- `crates/ajax-web/web/dist/app.css` (via `npm run web:build` only)
- `crates/ajax-web/web/dist/app.js` (via `npm run web:build` only)
- `crates/ajax-web/web/dist/index.html` (via `npm run web:build` only)

## 3. Forbidden changes

- Any Rust crate, API, projection, or backend file
- `TerminalRawView.svelte` terminal behavior / theme hardcodes (leave alone)
- Routing, polling, state.ts status derivation
- Renaming CSS variables globally unless required for the selection/attention split
- Broad layout rewrites, new components, new dependencies
- Changing mustard usage for attention/waiting/remediation tones

## 4. Architecture context

Web Cockpit is display-only over server projections. Color is presentation only. Tokens live in `styles.css`; component-scoped styles live in Svelte `<style>` blocks. Dist assets are generated by `npm run web:build`.

## 5. Code anchors

Current confusion (mustard used for both selection and attention):

```css
/* TaskList.svelte — active pill uses mustard (WRONG for selection) */
.project-pill.is-active {
  background: var(--mustard);
  border-color: var(--mustard);
  color: #1c1714;
}

/* TaskList.svelte — attention badge correctly uses mustard */
.pill-badge {
  background: var(--mustard);
  color: #1c1714;
}
```

```css
/* styles.css — New task is teal; Dashboard has no selected state */
.bottom-nav button[data-bottom-action="new-task"] {
  background: var(--teal);
  color: var(--ink);
}
```

App already sets `aria-current="page"` on the Dashboard bottom-nav button when on dashboard/project routes (`App.svelte`).

Reuse existing tokens: `--teal`, `--teal-bright`, `--teal-deep`, `--mustard`, `--mustard-bright`, `--paper`, `--ink`.

## 7. Production edit instructions

1. **TaskList.svelte** scoped CSS:
   - Change `.project-pill.is-active` from mustard → teal family (`background: var(--teal)`, `border-color: var(--teal)`, text color that stays readable — prefer `var(--ink)` or `#1c1714` only if contrast requires it; teal-on-dark with cream ink is fine).
   - Keep `.pill-badge` and `.section-head.attention*` on mustard.
   - Optional small scanability win: give `.task-row` a subtle `background: color-mix(in srgb, var(--tone-bg) 55%, var(--paper-tint))` or similar so status tint shows without breaking swipe-reveal opacity. If that risks action bleed-through, skip and leave opaque `--paper-tint`.

2. **styles.css**:
   - Add `.bottom-nav button[aria-current="page"]` (and optionally `[aria-current="true"]`) with teal selected styling distinct from the New-task filled button (e.g. teal border + soft teal-deep/tint background, not identical to New).
   - Do not remove New-task teal fill.

3. Run `npm run web:build` so dist matches source.

## 9. Acceptance criteria

- New tests fail before CSS changes, pass after.
- Active project pill is teal; attention badge / Needs-you chrome remain mustard.
- Bottom-nav current page has a teal selected style via `aria-current`.
- Full web test suite green.
- Dist rebuilt; no conflict markers.

## 10. Stop conditions

- Tests pass before any CSS change (wrong assertions).
- Need to edit Rust or TerminalRawView for the change to work.
- Diff exceeds ~400 lines or requires layout restructure.
- Unrelated tests fail for reasons outside this packet.
